# 운영 배포 가이드

## 프로파일

- `local`: 개발용. 시크릿은 `application-local.properties`(gitignore)에서 읽는다. 실행: `--spring.profiles.active=local`
- `prod`: 운영용. 시크릿은 파일이 아니라 **환경변수**로 주입한다. 실행: `--spring.profiles.active=prod`

## 시크릿(환경변수)

`application.properties`가 `${...}`로 참조하는 값들을 배포 환경에 설정한다.

- DB: `DB_TNS_ALIAS`, `DB_WALLET_DIR`, `DB_USERNAME`, `DB_PASSWORD`
- 외부 API: `KAKAO_JS_KEY`, `KAKAO_REST_KEY`, `DATAGOKR_SERVICE_KEY`, `SEOUL_OPENDATA_KEY`, `FTC_FRANCHISE_KEY`, `REB_RONE_KEY`, `NAVER_CLIENT_ID/SECRET`, `BIZINFO_SERVICE_KEY`, `KSTARTUP_SERVICE_KEY`, `KIPRIS_API_KEY`
- 결제/AI: `TOSS_CLIENT_KEY`, `TOSS_SECRET_KEY`, `GEMINI_API_KEY`, `GEMINI_API_NEWS_KEY`
- 관리자 보안: `ADMIN_IP_ALLOWLIST`, `ADMIN_TRUST_DEVICE_SECRET`(prod 필수, 미설정 시 기동 중단)

prod 프로파일은 세션 쿠키 `secure=true`를 강제하므로 HTTPS 뒤에 두어야 한다.

## 운영 DB 스키마 변경

Hibernate가 운영 DDL을 실행하지 않으므로 애플리케이션 배포 전에 담당자가 SQL을 적용한다.

- 신규 DB: `src/main/resources/db/schema.sql`을 한 번 실행한다.
- 기존 DB: `schema.sql`을 다시 실행하지 않고 `scripts/db`의 날짜순 변경 SQL만 실행한다.
- 변경 SQL은 `USER_TABLES`, `USER_TAB_COLUMNS`, `USER_INDEXES`, `USER_CONSTRAINTS`를 확인한 뒤 없는 오브젝트만 생성한다. 중간에 실패하면 원인을 처리한 뒤 같은 파일을 다시 실행한다.
- 배포 기록에는 SQL 파일명, Git 커밋, 적용 DB, 작업자, 적용 시각, 검증 결과를 남긴다.

### 적용 전 확인

운영 DB 복구 지점이나 백업을 먼저 확보한다. DDL 잠금을 피하기 위해 배포 시간에는 쓰기 요청과 배치 적재를 중단한다.

```sql
-- 증분 SQL이 전제로 하는 기존 테이블
SELECT TABLE_NAME
FROM USER_TABLES
WHERE TABLE_NAME IN ('MEMBER', 'ADMIN_USER', 'INQUIRY', 'PAYMENT_ORDER', 'BATCH_JOB_LOG')
ORDER BY TABLE_NAME;

-- 부분 유니크 인덱스 생성 전에 결과가 없어야 한다.
SELECT DATASET_CD, COUNT(*) AS RUNNING_COUNT
FROM BATCH_JOB_LOG
WHERE STATUS = 'RUNNING'
GROUP BY DATASET_CD
HAVING COUNT(*) > 1;
```

기존 테이블이 빠져 있으면 증분 SQL을 실행하지 말고 DB 기준 버전을 먼저 확인한다. 중복 RUNNING 행이 있으면 실제 실행 여부를 확인한 뒤 관리자 화면에서 정리한다.

### 적용 순서

현재 운영 기준 증분 파일은 `scripts/db/20260712_operational_schema.sql`이다.

1. 기존 테이블 컬럼 추가: `MEMBER.VERSION`, `ADMIN_USER.LOCKED_AT`, `INQUIRY.ANSWER_READ_AT`, 결제 주문 구독 부여 기간
2. 결제 주문 부여 기간 제약조건 추가
3. `SHEDLOCK`, `LOGIN_RATE_HIT`, `AI_REPORT_QUOTA` 생성
4. `SPRING_SESSION`, `SPRING_SESSION_ATTRIBUTES` 생성
5. `UX_BATCH_JOB_RUNNING`과 지원 인덱스 생성

PowerShell에서 지갑 경로와 접속 별칭을 설정한 뒤 실행한다. 비밀번호는 명령행에 넣지 않고 SQL*Plus 프롬프트에서 입력한다.

```powershell
$env:TNS_ADMIN = $env:DB_WALLET_DIR
sqlplus "$env:DB_USERNAME@$env:DB_TNS_ALIAS" "@scripts\db\20260712_operational_schema.sql"
```

`WHENEVER SQLERROR EXIT`가 설정돼 있으므로 종료 코드가 0인지 확인한 뒤 애플리케이션을 배포한다.

### 적용 후 검증

```sql
SELECT TABLE_NAME
FROM USER_TABLES
WHERE TABLE_NAME IN (
    'SHEDLOCK', 'LOGIN_RATE_HIT', 'AI_REPORT_QUOTA',
    'SPRING_SESSION', 'SPRING_SESSION_ATTRIBUTES'
)
ORDER BY TABLE_NAME;

SELECT TABLE_NAME, COLUMN_NAME, NULLABLE, DATA_DEFAULT
FROM USER_TAB_COLUMNS
WHERE (TABLE_NAME = 'MEMBER' AND COLUMN_NAME = 'VERSION')
   OR (TABLE_NAME = 'ADMIN_USER' AND COLUMN_NAME = 'LOCKED_AT')
   OR (TABLE_NAME = 'INQUIRY' AND COLUMN_NAME = 'ANSWER_READ_AT')
   OR (TABLE_NAME = 'PAYMENT_ORDER'
       AND COLUMN_NAME IN ('SUBSCRIPTION_STARTED_AT', 'SUBSCRIPTION_ENDED_AT'))
ORDER BY TABLE_NAME, COLUMN_NAME;

SELECT INDEX_NAME, UNIQUENESS, STATUS
FROM USER_INDEXES
WHERE INDEX_NAME IN (
    'UX_BATCH_JOB_RUNNING', 'IX_LOGIN_RATE_HIT',
    'SPRING_SESSION_IX1', 'SPRING_SESSION_IX2', 'SPRING_SESSION_IX3'
)
ORDER BY INDEX_NAME;

SELECT CONSTRAINT_NAME, STATUS, VALIDATED
FROM USER_CONSTRAINTS
WHERE CONSTRAINT_NAME = 'CK_PAY_SUB_PERIOD';
```

검증 후 `/actuator/health`가 `UP`인지 확인하고 로그인, 세션 유지, 배치 중복 실행 방지, AI 리포트 한도 선점을 확인한다.

### 실패와 롤백

Oracle DDL은 자동 커밋되므로 SQL 파일 전체가 하나의 트랜잭션으로 롤백되지 않는다. 적용 중 실패하면 이미 생성된 오브젝트를 임의로 삭제하지 말고 원인을 해결한 뒤 멱등 SQL을 재실행한다.

애플리케이션 롤백 시 이번 변경은 모두 추가형이라 컬럼과 테이블을 그대로 둘 수 있다. 오브젝트 제거가 꼭 필요하면 애플리케이션을 중지하고 데이터 보존 여부를 확인한 뒤 별도 승인된 역방향 SQL이나 DB 복구 지점을 사용한다. 특히 `SPRING_SESSION` 계열 테이블은 실행 중 삭제하지 않는다.

## 운영 관측

- 헬스체크: `GET /actuator/health` (LB/업타임 모니터용, UP/DOWN만 노출)
- 로그: 콘솔 + `logs/app.log`(롤링, 10MB/14일/500MB 상한). `LOG_DIR` 환경변수로 경로 변경 가능.

## 배포 동작

- 무중단 종료: `server.shutdown=graceful` (진행 중 요청을 마치고 내림, 최대 20초)
- 세션: 유휴 30분 만료(회원 자동로그인 선택 시 14일). Spring Session JDBC로 DB(`SPRING_SESSION`)에 영속화하므로 재배포·재시작에도 로그인이 유지되고 다중 인스턴스에서 세션이 공유된다(스티키 세션 불필요).

## 자동 정리(스케줄)

- 배치 실행 이력(`BATCH_JOB_LOG`)은 매일 새벽 4시(KST) 90일 이전 이력을 정리한다.
- 감사 로그(`AUDIT_LOG`)는 컴플라이언스 대상이라 자동 정리하지 않는다(보존 정책은 팀 결정).

## 데이터 적재

- 앱 적재는 관리자 콘솔 > 데이터 적재에서 수동 트리거한다(자동 스케줄 없음).
- 진행 중 상태가 멈춰 재적재가 막히면 해당 데이터셋 모달의 "진행 중 상태 초기화"로 푼다(재시작 시 자동 정리도 됨).
